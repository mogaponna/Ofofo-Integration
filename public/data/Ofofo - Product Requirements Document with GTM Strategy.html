2. PRODUCT OVERVIEW

2.1 Product Description

Ofofo.ai is an agentic AI platform that automates security due diligence processes between Large Enterprises and their Target Entity vendors. The platform addresses critical gaps in existing solutions by:

  • Saving 10-15 hours of CISO's manual verifications through intelligent AI agents, transforming investigators into human-in-the-loop reviewers instead of manual investigators
  • Automating evidence collection and validation from Target Entity vendors
  • Providing real-time security posture monitoring and assessment
  • Offering continuous compliance tracking and reporting
  • Enabling Target Entities to demonstrate enterprise-grade security without massive overhead

The platform serves as an intelligent intermediary that understands both enterprise security requirements and Target Entity operational realities, bridging the gap through AI-powered automation and continuous monitoring.

2.2 Current State Assessment

Core Product: Questionnaire AI System

Ofofo.ai has developed a sophisticated Questionnaire AI that revolutionizes security due diligence through advanced agentic workflows. The system operates through a multi-tier architecture that ensures fast, accurate, and comprehensive responses to due diligence questions.

4.3 Core Use Cases

Use Case 1: Existing Target Entity Customer Due Diligence Response

Trigger: Enterprise requests security assessment of Target Entity vendor who is already an Ofofo customer

Process:

  1. Enterprise sends security questionnaire via SaaS portals (CyberGRX) or Excel forms
  2. Target Entity receives due diligence request
  3. Target Entity uses Questionnaire AI to generate responses based on existing Knowledge Graphs and data room
  4. AI agents perform analysis against 1400+ controls using current security posture
  5. For any gaps identified, AI provides options: implement missing controls, hire fractional support, or provide justified "No" responses
  6. Internal auditor (fractional CISO) validates AI-generated assessment if required
  7. Target Entity submits completed questionnaire with evidence to Enterprise within 24-hour target
Use Case 2: Target Entity Compliance Journey

Trigger: Target Entity wants to achieve specific compliance frameworks

Process:

  1. AI agent asks Target Entity about their ideal customers (e.g., "United Health Group")
  2. Based on customer profile, agent intelligently suggests required compliance frameworks (Healthcare + AI = SOC2 + HIPAA minimum)
  3. Platform maps current security posture against required controls
  4. AI identifies gaps and recommends improvement actions with guided implementation
  5. Target Entity implements controls with AI providing guided tours and required implementation steps (similar to Cursor's early approach)
  6. AI provides test cases to users for validation - users execute tests and share results/errors with AI for consultative troubleshooting
  7. Agent checks if implementation is proper by analyzing test results and confirms evidence can be collected
  8. Platform tracks progress and maintains evidence repository with continuous monitoring (read-only access only)
  9. If Target Entity has internal audit capability, they can self-audit; otherwise, they can hire fractional auditors from Ofofo's procurement AI for Human-in-the-Loop validation
  10. External auditor certifies compliance when ready

Critical Security Boundary: Ofofo.ai never has edit or write access to subprocessors. It only provides guided tours, required steps, test cases, and consultative guidance. Users implement everything themselves and share feedback/errors for AI troubleshooting support.

Use Case 3: Security Procurement Marketplace

Trigger: Target Entity needs security products or services to address due diligence requirements (compliance gaps, enterprise-specific requirements like "data at rest encryption," or any DD question requirements)

Process:

  1. Target Entity identifies need from DD question analysis (e.g., "Do you have data at rest encryption?" - not required for ISO27001 but asked by enterprise)
  2. Target Entity provides budget constraints and implementation timeline requirements
  3. Procurement AI first searches existing vendor/product options that match budget and timeline criteria
  4. If suitable matches found, Target Entity can directly procure:
    • Product licenses directly (e.g., Acronis license through Acronis)
    • Implementation support separately (e.g., AxiomIO for installation and ongoing support)
    • Combined product + service packages from marketplace vendors
  5. If no suitable matches found OR Target Entity is not satisfied with existing options, then Target Entity creates RFQ specifying:
    • Required security controls or DD question requirements
    • Budget range and timeline constraints
    • Preferred implementation approach (product only, service only, or combined)
  6. Qualified security companies receive RFQ notifications and submit proposals
  7. Vendors map their proposals to specific controls and DD requirements
  8. Gamified scoring system ranks vendors based on fit, value, budget, and timeline
  9. Target Entity selects vendor/product and tracks implementation progress

Example Flow: Enterprise asks "Do you have data at rest encryption?" → Target Entity sets budget ($5,000) and timeline (2 weeks) → Procurement AI finds 3 existing encryption solutions within budget → If satisfied, direct procurement; if not, create RFQ for custom requirements.

Use Case 4: Enterprise Consent-Based Due Diligence (Existing Ofofo Customers)

Trigger: Enterprise needs assessment of existing Ofofo Target Entity customer and wants direct platform access

Process:

  1. Enterprise requests due diligence consent from Target Entity
  2. Target Entity approves consent through platform
  3. Enterprise instantly accesses current risk scores and reports
  4. Real-time monitoring provides ongoing risk updates
  5. Automated alerts notify of any security posture changes
Use Case 5: Independent Audit Certification

Trigger: External auditor needs to certify Target Entity compliance

Process:

  1. External auditor accesses Target Entity's compliance documentation
  2. Platform provides standardized audit templates and reports
  3. Auditor can ask/comment for additional information from Target Entity
  4. Ofofo.ai answers auditor questions using Questionnaire AI; if AI cannot provide required evidence, Target Entity's internal auditors upload information to data room
  5. Platform helps CPAs with limited cybersecurity knowledge gain confidence to attest SOC2 reports through guided audit processes
  6. Auditor manages surveillance audit schedules, peer review requirements, and attestation risks
  7. Final certification integrates platform data with auditor insights
  8. When compliance status or risk scores are updated, automatic notifications are sent to all Enterprises currently working with the Target Entity

Golden Opportunity: Enabling untapped CPAs to confidently attest SOC2 reports despite limited cybersecurity knowledge, expanding the auditor pool significantly.

9. SUCCESS METRICS & KPIs

9.1 The North Star Metric for Agentic AI Systems

Core Philosophy: Traditional software metrics (revenue, users, uptime) are necessary but insufficient for measuring agentic AI success. An agentic system's true value lies in its ability to autonomously complete complex workflows that would otherwise require human expertise and time. The quality of autonomous execution directly drives all business outcomes.

Agent Effectiveness Score (AES)

Definition: A composite metric that measures how effectively the AI agents complete end-to-end workflows autonomously, with minimal human intervention, while maintaining quality and user trust.

Formula Components

The Agent Effectiveness Score (AES) is calculated as:

AES = (Autonomy × Accuracy × Velocity × Trust) ÷ Intervention_Burden

Where:
- Autonomy = % of tasks completed without human intervention
- Accuracy = % of AI outputs accepted without modification  
- Velocity = (Baseline_Time ÷ Actual_Time) normalized to 0-1 scale
- Trust = Net Promoter Score (NPS) ÷ 100 (normalized to 0-1)
- Intervention_Burden = (Human_Time_Required ÷ Total_Time_Saved) + 1

Final AES is expressed as a score from 0-100

Component Definitions & Measurement

1. Autonomy Score

What it Measures: The percentage of workflows the AI completes from start to finish without requiring human intervention.

How to Measure Through Product:

  • Track Workflow Initiation: Log when an AI agent begins a workflow (e.g., questionnaire received, compliance gap analysis started)
  • Track Human Touchpoints: Log every time a human manually intervenes:
    • Manual answer override in questionnaire
    • Human-in-the-loop approval before submission
    • Manual evidence upload prompted by AI
    • Manual control implementation instead of automated
  • Track Workflow Completion: Log when workflow reaches final state

Formula:

Autonomy = (Total_Workflows_Completed - Workflows_Requiring_Intervention) ÷ Total_Workflows_Completed × 100

Example:
- 100 questionnaires processed
- 85 submitted without any human intervention
- Autonomy Score = (100 - 15) ÷ 100 = 85%

Database Schema Example:

Table: workflow_events
- workflow_id (UUID)
- workflow_type (enum: questionnaire, compliance_check, gap_analysis, etc.)
- status (enum: started, in_progress, human_intervention_required, completed)
- intervention_type (enum: null, manual_edit, approval_needed, missing_data, etc.)
- timestamp (datetime)
- user_id (UUID, null if AI action)

Query for Autonomy:
SELECT 
    (COUNT(DISTINCT workflow_id) FILTER (WHERE status = 'completed' AND intervention_type IS NULL)) /
    COUNT(DISTINCT workflow_id) FILTER (WHERE status = 'completed')) * 100 AS autonomy_score
FROM workflow_events
WHERE workflow_type = 'questionnaire'
  AND created_at >= CURRENT_DATE - INTERVAL '30 days';
2. Accuracy Score

What it Measures: The percentage of AI-generated outputs that humans accept without modification.

How to Measure Through Product:

  • Track AI-Generated Outputs: Every answer, recommendation, control suggestion AI produces
  • Track Human Modifications: Log when humans edit AI outputs:
    • Minor edits (< 10% of content changed)
    • Major edits (10-50% changed)
    • Complete rewrites (> 50% changed)
    • Accepted as-is (0% changed)
  • Weight by Edit Severity: Complete rewrites count more heavily against accuracy than minor edits

Formula:

Accuracy = [(Accepted_As_Is × 1.0) + (Minor_Edit × 0.8) + (Major_Edit × 0.5) + (Complete_Rewrite × 0)] ÷ Total_Outputs × 100

Example:
- 100 AI-generated questionnaire answers
- 70 accepted as-is (70 × 1.0 = 70 points)
- 20 minor edits (20 × 0.8 = 16 points)
- 7 major edits (7 × 0.5 = 3.5 points)
- 3 complete rewrites (3 × 0 = 0 points)
- Accuracy = (70 + 16 + 3.5 + 0) ÷ 100 = 89.5%

Database Schema Example:

Table: ai_outputs
- output_id (UUID)
- workflow_id (UUID)
- output_type (enum: answer, recommendation, control_suggestion, etc.)
- original_content (text)
- final_content (text)
- edit_type (enum: accepted, minor_edit, major_edit, rewrite)
- similarity_score (float, 0-1, calculated via text similarity)
- timestamp (datetime)

Query for Accuracy:
SELECT 
    (SUM(CASE 
        WHEN edit_type = 'accepted' THEN 1.0
        WHEN edit_type = 'minor_edit' THEN 0.8
        WHEN edit_type = 'major_edit' THEN 0.5
        WHEN edit_type = 'rewrite' THEN 0
    END) / COUNT(*)) * 100 AS accuracy_score
FROM ai_outputs
WHERE created_at >= CURRENT_DATE - INTERVAL '30 days';
3. Velocity Score

What it Measures: How much faster the AI completes tasks compared to manual baseline.

How to Measure Through Product:

  • Establish Baselines: Measure time for humans to complete tasks manually:
    • Questionnaire: Average 48-72 hours (based on industry data)
    • Gap Analysis: Average 2-3 weeks
    • Control Implementation Research: Average 8-16 hours per control
  • Track AI Completion Time: Log timestamps from workflow start to completion
  • Normalize to 0-1 Scale: Use logarithmic scaling to account for diminishing returns

Formula:

Raw_Velocity_Ratio = Baseline_Time ÷ AI_Completion_Time
Velocity_Score = MIN(1.0, LOG10(Raw_Velocity_Ratio + 1) ÷ LOG10(11))

Example for Questionnaire:
- Baseline: 60 hours (industry average)
- AI Completion: 0.25 hours (15 minutes)
- Raw_Velocity_Ratio = 60 ÷ 0.25 = 240x faster
- Velocity_Score = LOG10(241) ÷ LOG10(11) = 2.38 ÷ 1.04 = ~0.95 (on 0-1 scale)

Converting to 0-100 scale: 0.95 × 100 = 95%

Database Schema Example:

Table: workflow_timing
- workflow_id (UUID)
- workflow_type (enum)
- started_at (datetime)
- completed_at (datetime)
- duration_seconds (integer, calculated)
- baseline_seconds (integer, from workflow_baselines table)

Table: workflow_baselines
- workflow_type (enum, primary key)
- baseline_seconds (integer)
- last_updated (datetime)

Query for Velocity:
WITH velocity_calcs AS (
    SELECT 
        workflow_type,
        baseline_seconds::float / NULLIF(duration_seconds, 0) AS velocity_ratio
    FROM workflow_timing
    WHERE completed_at >= CURRENT_DATE - INTERVAL '30 days'
      AND duration_seconds > 0
)
SELECT 
    AVG(LEAST(1.0, LOG(velocity_ratio + 1) / LOG(11))) AS normalized_velocity
FROM velocity_calcs;
4. Trust Score

What it Measures: User confidence in the AI system, measured through Net Promoter Score (NPS).

How to Measure Through Product:

  • In-App NPS Survey: After significant milestones (e.g., first questionnaire completion, SOC2 certification achieved)
  • Question: "How likely are you to recommend Ofofo.ai to a colleague?" (0-10 scale)
  • Calculate NPS: % Promoters (9-10) - % Detractors (0-6)
  • Normalize: Divide by 100 to get 0-1 scale for formula

Formula:

NPS = (% Promoters - % Detractors)
Trust_Score = NPS ÷ 100

Example:
- 100 responses
- 60 Promoters (60%)
- 30 Passives (30%)
- 10 Detractors (10%)
- NPS = 60% - 10% = 50
- Trust_Score = 50 ÷ 100 = 0.5

Trust Score on 0-100 scale: 50%

Database Schema Example:

Table: nps_surveys
- survey_id (UUID)
- user_id (UUID)
- score (integer, 0-10)
- workflow_id (UUID, nullable)
- milestone_type (enum: first_questionnaire, certification_achieved, etc.)
- feedback_text (text, nullable)
- created_at (datetime)

Query for Trust Score:
WITH nps_calc AS (
    SELECT 
        COUNT(*) FILTER (WHERE score >= 9) * 100.0 / COUNT(*) AS promoter_pct,
        COUNT(*) FILTER (WHERE score <= 6) * 100.0 / COUNT(*) AS detractor_pct
    FROM nps_surveys
    WHERE created_at >= CURRENT_DATE - INTERVAL '90 days'
)
SELECT 
    (promoter_pct - detractor_pct) AS nps_score,
    (promoter_pct - detractor_pct) / 100.0 AS trust_score_normalized
FROM nps_calc;
5. Intervention Burden

What it Measures: The "cost" of human intervention relative to time saved. Penalizes systems that require significant human oversight despite appearing autonomous.

How to Measure Through Product:

  • Track Human Time Spent: When humans intervene, log time spent:
    • Time reviewing AI outputs
    • Time making edits
    • Time gathering missing evidence AI requested
    • Time troubleshooting issues
  • Track Total Time Saved: Baseline Time - AI Completion Time

Formula:

Intervention_Burden = (Human_Time_Required ÷ Total_Time_Saved) + 1

The "+1" ensures the burden is always ≥ 1, preventing division by zero and ensuring 
systems with zero intervention have burden of 1 (no penalty).

Example 1 - Low Burden:
- Baseline: 60 hours
- AI Time: 0.25 hours
- Human Review Time: 0.5 hours
- Time Saved: 60 - 0.25 = 59.75 hours
- Intervention_Burden = (0.5 ÷ 59.75) + 1 = 1.008

Example 2 - High Burden:
- Baseline: 60 hours
- AI Time: 5 hours
- Human Intervention: 10 hours (extensive editing, missing evidence gathering)
- Time Saved: 60 - 5 = 55 hours
- Intervention_Burden = (10 ÷ 55) + 1 = 1.18

Database Schema Example:

Table: human_interventions
- intervention_id (UUID)
- workflow_id (UUID)
- user_id (UUID)
- intervention_type (enum: review, edit, evidence_gathering, troubleshooting)
- time_spent_seconds (integer)
- timestamp (datetime)

Query for Intervention Burden:
WITH time_calcs AS (
    SELECT 
        wt.workflow_id,
        (wb.baseline_seconds - wt.duration_seconds) AS time_saved_seconds,
        COALESCE(SUM(hi.time_spent_seconds), 0) AS human_time_seconds
    FROM workflow_timing wt
    JOIN workflow_baselines wb ON wt.workflow_type = wb.workflow_type
    LEFT JOIN human_interventions hi ON wt.workflow_id = hi.workflow_id
    WHERE wt.completed_at >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY wt.workflow_id, wb.baseline_seconds, wt.duration_seconds
)
SELECT 
    AVG((human_time_seconds::float / NULLIF(time_saved_seconds, 0)) + 1) AS intervention_burden
FROM time_calcs
WHERE time_saved_seconds > 0;

Calculating the Final AES

Step-by-Step Calculation:

  1. Gather all component scores from database queries (each on 0-100 or 0-1 scale)
  2. Normalize all scores to 0-1 scale (if not already)
  3. Calculate: AES_raw = (Autonomy × Accuracy × Velocity × Trust) ÷ Intervention_Burden
  4. Scale to 0-100: AES_final = AES_raw × 100

Example Calculation:

Component Values (from examples above):
- Autonomy = 0.85 (85%)
- Accuracy = 0.895 (89.5%)
- Velocity = 0.95 (95%)
- Trust = 0.50 (NPS of 50)
- Intervention_Burden = 1.008

AES_raw = (0.85 × 0.895 × 0.95 × 0.50) ÷ 1.008
AES_raw = 0.361 ÷ 1.008
AES_raw = 0.358

AES_final = 0.358 × 100 = 35.8

**Interpretation: An AES of 35.8 indicates a moderately effective agentic system 
with room for improvement, particularly in Trust (NPS 50 is average).**

Complete Database Query for AES:

WITH metrics AS (
    -- Autonomy
    SELECT 
        (COUNT(DISTINCT workflow_id) FILTER (WHERE status = 'completed' AND intervention_type IS NULL))::float /
        NULLIF(COUNT(DISTINCT workflow_id) FILTER (WHERE status = 'completed'), 0) AS autonomy
    FROM workflow_events
    WHERE created_at >= CURRENT_DATE - INTERVAL '30 days'
),
accuracy AS (
    SELECT 
        (SUM(CASE 
            WHEN edit_type = 'accepted' THEN 1.0
            WHEN edit_type = 'minor_edit' THEN 0.8
            WHEN edit_type = 'major_edit' THEN 0.5
            WHEN edit_type = 'rewrite' THEN 0
        END) / NULLIF(COUNT(*), 0)) AS accuracy_score
    FROM ai_outputs
    WHERE created_at >= CURRENT_DATE - INTERVAL '30 days'
),
velocity AS (
    SELECT 
        AVG(LEAST(1.0, LOG(baseline_seconds::float / NULLIF(duration_seconds, 0) + 1) / LOG(11))) AS velocity_score
    FROM workflow_timing
    WHERE completed_at >= CURRENT_DATE - INTERVAL '30 days'
),
trust AS (
    SELECT 
        ((COUNT(*) FILTER (WHERE score >= 9) * 100.0 / NULLIF(COUNT(*), 0)) -
         (COUNT(*) FILTER (WHERE score <= 6) * 100.0 / NULLIF(COUNT(*), 0))) / 100.0 AS trust_score
    FROM nps_surveys
    WHERE created_at >= CURRENT_DATE - INTERVAL '90 days'
),
burden AS (
    SELECT 
        AVG((COALESCE(SUM(hi.time_spent_seconds), 0)::float / 
             NULLIF(wb.baseline_seconds - wt.duration_seconds, 0)) + 1) AS intervention_burden
    FROM workflow_timing wt
    JOIN workflow_baselines wb ON wt.workflow_type = wb.workflow_type
    LEFT JOIN human_interventions hi ON wt.workflow_id = hi.workflow_id
    WHERE wt.completed_at >= CURRENT_DATE - INTERVAL '30 days'
    GROUP BY wt.workflow_id, wb.baseline_seconds, wt.duration_seconds
)
SELECT 
    m.autonomy,
    a.accuracy_score,
    v.velocity_score,
    t.trust_score,
    b.intervention_burden,
    ((m.autonomy * a.accuracy_score * v.velocity_score * t.trust_score) / 
     NULLIF(b.intervention_burden, 0)) * 100 AS agent_effectiveness_score
FROM metrics m
CROSS JOIN accuracy a
CROSS JOIN velocity v
CROSS JOIN trust t
CROSS JOIN burden b;
